# Decision Log

| # | Decision | Rationale | Alternatives considered |
|---|---|---|---|
| 1 | Compile bench + tests with `tsc`, run as `.js`, instead of `--experimental-strip-types` | Sandbox Node is 20.20.1; task brief explicitly names this as the fallback for older Node | Requiring Node 22.6+ (rejected: not available in this environment) |
| 2 | Hand-roll BM25, Levenshtein, boolean parser, tokenizer | Zero-runtime-dependency constraint leaves no lower rung on the complexity-brake ladder | Using a search library (rejected: violates zero-dep constraint), using stemmer/NLP package (rejected: not needed, identity stemmer is spec default) |
| 3 | Support parenthesized grouping in the boolean string parser even though not explicitly spec'd | The "deeply nested AND/OR/NOT" edge case is ambiguous without an explicit grouping token; parens are the minimal, standard mechanism and keep operator precedence testable/unambiguous | Relying purely on fixed precedence without grouping (rejected: cannot express arbitrary nesting requested by the edge-case list) |
| 4 | Phrase proximity formula: BM25 term score * (1 + proximityBonus), proximityBonus = 1 / (1 + averageGap - phraseLength + 1) clamped, computed only over in-order, matching-slop windows | Spec only requires "closer term positions rank higher," not an exact formula; documented explicitly so reviewers can evaluate the choice | Binary match/no-match without gradation (rejected: spec explicitly wants proximity-sensitive ranking) |
| 5 | `stopwords: 'english'` uses a fixed ~30-term hardcoded list (a, an, and, the, etc.) | No runtime dependency permitted for a stopword dataset; a small curated list satisfies "all-stopword query" edge case without pulling in a package | Using a large external stopword list (rejected: would require a dependency or a large embedded asset disproportionate to test needs) |
| 6 | `index_memory_estimate_kb` computed as `JSON.stringify(toJSON()).length / 1024` | Deterministic, dependency-free, cross-platform proxy for memory footprint; `process.memoryUsage()` is available but noisy/non-deterministic and includes unrelated heap state | `process.memoryUsage().heapUsed` delta (rejected: non-deterministic, polluted by GC timing) |
| 7 | Snippet extraction operates on raw stored field text (not tokenized form) | Preserves original casing/punctuation in output, matching typical search-UI snippet expectations and the `<mark>` example in the spec | Snippet built from token array only (rejected: would lose original formatting) |
| 8 | Filter applied after ranking/sorting, before offset/limit slicing | Matches spec wording "post-filter after ranking" literally | Filtering before scoring (rejected: contradicts explicit spec wording) |
| 9 | Bare `NOT` immediately after a WORD/AND chain continues the same conjunctive clause instead of starting a new implicit-OR atom (`"a AND b NOT c"` parses as `a AND b AND (NOT c)`) | Matches the spec's own literal example ("docs with both a and b and without c"); the alternative (treating NOT as starting a new OR-joined atom) silently dropped the NOT clause and was caught by a validation-contract test | Treating every adjacent atom start (including NOT) as implicit OR (rejected: contradicts the spec's worked example; caught during VERIFY) |
| 10 | CJK text with no whitespace tokenizes as one contiguous token per run (no CJK word segmentation implemented) | `\p{L}\p{N}` correctly identifies CJK characters as letters but cannot infer word boundaries without a segmentation algorithm/dictionary, which is a distinct, much larger problem and would require either a large embedded dictionary or a dependency — out of scope for "unicode-safe tokenizer," which the spec's own example (`搜索`, a bare 2-character word) is consistent with | Shipping a CJK segmentation algorithm (rejected: disproportionate scope, likely requires a dependency or large embedded frequency dictionary to be accurate) |
| 11 | Removed an overly aggressive result filter (`score <= 0 && matchedByField.size === 0`) in `SearchIndex.search` that was silently dropping legitimate NOT-query matches (score 0, no matched terms by design) | Found via a validation-contract test (nested `or`/`and`/`not` structured query) that returned too few docs; every entry already present in the `evaluateQuery` match map is a real match, so no extra filter is needed — the bug was defensive code that contradicted the query engine's own contract | Keeping the filter and instead tagging NOT-results with a synthetic nonzero score (rejected: distorts ranking and score semantics for no benefit) |
| 12 | Fixed a O(vocabulary-size) performance bug in `PostingsStore.clearField`: it originally scanned every term in the whole index on every single `indexField` call (including first-time inserts with nothing to clear), causing ~6s/10k-doc build times and dominating GC. Added a reverse index (`docFieldTerms`: field -> docId -> Set<term>) so clearing only touches the terms that specific (docId, field) previously used. | Found via `node --prof` profiling of the benchmark build step, which showed `clearField` as the dominant JS hot path; the fix is O(terms actually used by that doc/field) instead of O(total distinct terms in the corpus) | Leaving the O(vocab) scan and just accepting slower builds (rejected: benchmark is a required deliverable and must be representative of real usage, not an artifact of an accidental full-index scan on every add) |
| 13 | Enlarged the benchmark's synthetic vocabulary from ~800 to 20,000 deterministic CVCV/CVCVCV words | With a real corpus this size, an 800-word vocabulary gives each term a document frequency in the thousands, which is unrealistic and made every query touch a large fraction of the corpus, hiding the benefit of the inverted index and dominating the benchmark JSON with GC/allocation noise rather than representative query cost | Keeping the original 800-word vocabulary (rejected: not representative of real full-text search workloads, where terms are far more distinct) |
